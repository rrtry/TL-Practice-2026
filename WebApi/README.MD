# WebApi2

## Задача

Ваша задача - создать приложение на ASP.NET Core WebApi для бронирования и управления средствами размещения (отели и др.).

В рамках решения реализуйте два API:

## 1. PropertiesApi

Это API для управления объектами размещения и их категориями.

### Функционал PropertiesApi:

- CRUD по Property (средствам размещения)
  - Create: создать новый отель.
  - Read: получить список отелей и отдельный отель по ID.
  - Update: редактировать сведения об отеле.
  - Delete: удалить объект размещения.
- CRUD по RoomType (категориям номеров)
  - Create: добавить категорию/тип номера (RoomType) к Property.
  - Read: получить все RoomType для Property или один RoomType по ID.
  - Update: изменить параметры RoomType.
  - Delete: удалить RoomType.

Каждый вид CRUD-операций реализуйте отдельными методами контроллеров, соответствующими методам HTTP (GET, POST, PUT/PATCH, DELETE).

Структура контроллеров:
- Например, PropertiesController для Property, RoomTypesController для RoomType (может быть дочерним по отношению к Property).

## 2. ReservationApi

API для поиска, создания и управления бронированиями.

### Функционал ReservationApi:

- **Поиск вариантов бронирования**
  - Поиск доступных вариантов на основе:
    - Город (City)
    - Период дат (ArrivalDate, DepartureDate)
    - Количество гостей (Min/MaxPersonCount)
    - Макс цена
  - Результат - список Property и RoomType, подходящих под эти параметры и доступных на указанный период.

- **Создание бронирования**
  - Выбор Property и RoomType, указание количества гостей, дат заезда/выезда.
  - Проверка доступности на период.
  - Расчет стоимости бронирования (Total) по актуальной цене на момент создания и длительности (число ночей).
  - Создание нового Reservation.

- **Просмотр бронирований**
  - Получение списка всех бронирований (по фильтрам: по property, по датам, по гостю и пр.) или бронирования по ID.

- **Отмена бронирования**
  - В случае запроса на отмену найти Reservation по ID и отметить его как отмененный или удалить (зависит от выбранной логики).

Реализация должна полностью соответствовать REST-подходу (четко разделенные по ресурсу маршруты и HTTP-методы).

## Описание сущностей

Описаны основные сущности (Entities), которые должны быть реализованы в модели вашего приложения:

### 1. Property

Property - средство размещения (отель, апартаменты и др.).

- Id - идентификатор (GUID или Int)
- Name - название (строка)
- Country - страна
- City - город
- Address - адрес
- Latitude - географическая широта (decimal/double)
- Longitude - географическая долгота (decimal/double)

### 2. RoomType

RoomType - категория номера.

- Id
- PropertyId - ссылка на Property
- Name
- DailyPrice - цена за ночь
- Currency
- MinPersonCount - минимальное число гостей
- MaxPersonCount - максимальное число гостей
- Services - перечень сервисов (можете выбрать структуру: строка/enum/список)
- Amenities - перечень удобств (аналогично)

### 3. Reservation

Reservation - бронирование.

- Id
- PropertyId - объект размещения
- RoomTypeId - тип номера
- ArrivalDate - дата прибытия
- DepartureDate - дата выезда
- ArrivalTime - время заезда
- DepartureTime - время отъезда
- GuestName - ФИО гостя
- GuestPhoneNumber
- Total - стоимость бронирования (рассчитывается при создании: цена за 1 ночь × число ночей)
- Currency

**Важно:** При расчёте стоимости учитывайте, что цена категории может меняться между разными датами, но для упрощения задачи используйте на момент бронирования актуальную цену.

## Требования к архитектуре и технологиям

1. REST API - все эндпоинты проектируйте согласно RESTful-принципам.
2. DI (Dependency Injection) - для всех сервисов, репозиториев и вспомогательных классов.
3. Паттерн Repository - для Property, RoomType, Reservation. Вынесите интерфейсы и реализации в отдельные слои.
4. **Слоями:**
   - Domain: core-логика и сущности, интерфейсы сервисов и репозиториев;
   - Infrastructure: имплементации (например, InMemory/DB);
   - WebApi: контроллеры, DTO, настройки.
5. Swagger - для интерактивной документации.

## Примеры маршрутов/методов (endpoint-ы)

### PropertiesApi

- `GET /api/properties` - список всех Properties
- `GET /api/properties/{id}` - подробности Property
- `POST /api/properties` - добавить Property
- `PUT /api/properties/{id}` - обновить Property
- `DELETE /api/properties/{id}` - удалить Property
- `GET /api/properties/{propertyId}/roomtypes` - получить RoomTypes Property
- `GET /api/roomtypes/{id}` - получить RoomType
- `POST /api/properties/{propertyId}/roomtypes` - добавить RoomType
- `PUT /api/roomtypes/{id}` - обновить RoomType
- `DELETE /api/roomtypes/{id}` - удалить RoomType

### ReservationApi

- `GET /api/search?city=...&arrivalDate=...&departureDate=...&guests=...` - поиск вариантов размещения
- `POST /api/reservations` - создать бронь (требует PropertyId, RoomTypeId, ArrivalDate, DepartureDate, Guests и т.д.)
- `GET /api/reservations` - список всех бронирований с поддержкой фильтрации
- `GET /api/reservations/{id}` - получение информации о конкретном бронировании
- `DELETE /api/reservations/{id}` - отмена бронирования

## Рекомендации к логике

- Поиск вариантов - учитывайте занятость номеров (RoomType) за выбранный период, соответствие количества гостей допускам RoomType и параметрам города.
- Создание бронирования - не разрешайте создавать бронь, если нет доступных мест или параметры не подходят (валидация!).
- Стоимость бронирования = DailyPrice текущего RoomType × количество ночей.
- Отмена - реализуйте через физическое удаление или логическое (например, флаг Canceled, в зависимости от архитектуры).

## FAQ (ключевые моменты)

1. **Нужно ли возвращать сущности напрямую?**  
   Нет! Используйте DTO для входных/выходных моделей.

2. **Какой должна быть структура проекта?**  
   - `/Domain` (сущности, интерфейсы, бизнес-правила)
   - `/Infrastructure` (реализации хранилищ, мапперы)
   - `/WebApi` (контроллеры, DTO, startup, настройки DI, Swagger)

3. **Как реализовать «доступность» RoomType?**  
   Для каждого RoomType храните info о количестве доступных мест и проверяйте бронирования на пересекающиеся даты.

4. **Где делать бизнес-валидацию?**  
   В сервисах бизнес-логики (не в контроллере!).
